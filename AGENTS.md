# AGENTS.md

Guidance for AI coding agents working in this repository.

## What this is

Wordweave — a word-search puzzle game with English/Ukrainian support, themed word categories, and a difficulty (grid size) picker. Plain HTML/CSS/JS single-page app, no framework, no build step, no package manager. Everything runs directly in the browser.

## Running & verifying changes

There is no build step. Serve the directory over HTTP and load it in a browser:

```bash
python3 -m http.server 8972
```

Port 8972 is this game's fixed dev-server port, tracked in the root [`AGENTS.md`](../AGENTS.md#local-dev-server-ports) alongside every other game's — don't change it or reuse it for another game in this repo. (This game was briefly, incorrectly, run on 8971 — `worder`'s port — during development; if you see stale references to 8971 anywhere in this project's history, that's what happened.)

`script.js` fetches `words.en.json` / `words.uk.json` at startup (whichever is active), so the app must be served over HTTP(S) — it will not work opened as a `file://` URL.

After editing `script.js`, sanity-check syntax before assuming it works:

```bash
node --check script.js
```

There are no automated tests. Verify behavior by loading the page in a browser (or headlessly via Playwright/`chromium-cli`, which this project's history has used for exactly this) and playing through the affected flow: generate a puzzle, trace a word forwards/diagonally/reversed by tapping cells in sequence, confirm badges appear in order and the last-tapped cell undoes on re-tap, confirm a completed word auto-marks found (cell highlight + struck-through chip) and updates the header's found-count badge, finish a whole puzzle and check the win message + stats, switch difficulty/language mid-puzzle (confirm modal should fire since progress exists, and the *same* category/subcategory should carry over), toggle Category to Select mode and confirm New Game opens the picker (including an Animals subcategory row and its "All Mixed" row) instead of generating directly, reload mid-puzzle (state — including category/subcategory — should resume, along with the timer), and open the Stats modal.

## Architecture

- **`index.html`** — static markup only: header (title + difficulty badge + found-count badge + timer + the "⋮" menu with Stats/Category-mode/Difficulty/Language), the board grid, the word-list panel, and four modals (`help-modal`, `stats-modal`, `confirm-modal`, `category-select-modal`). All modals share the `.modal`/`.modal-content` CSS classes and the same open/close pattern (toggle `.hidden`) — same pattern as the sibling `worder` project. User-facing text carries `data-i18n="key"` (sets `textContent`) or `data-i18n-title="key"` (sets `title` + `aria-label`); `applyStaticTranslations()` in `script.js` walks these on load and on language switch. Adding new UI text means adding both the attribute here and the key in `i18n.js` for every language.
- **`style.css`** — all styling. Theme colors are CSS custom properties on `:root`, overridden in `@media (prefers-color-scheme: dark)`. `#board` sizes itself from the `--cols` custom property (set in JS via `style.setProperty`) referenced in `grid-template-columns`/`grid-template-rows` — don't hardcode a column count in CSS, the same stylesheet serves all three grid sizes.
- **`i18n.js`** — `const I18N = { en: {...}, uk: {...} }`, loaded before `script.js`. Category labels (`categoryAnimals`, etc.) and difficulty labels (`levelEasy`, etc.) live here, not in the word JSON files — the JSON only holds the raw word lists (subcategory *names*, unlike category names, live directly in the word JSON — see below, they're not `i18n.js` keys). Keep both language objects in sync; a key present in one but not the other renders `undefined` for the missing language (no fallback).
- **`script.js`** — a single IIFE, no modules/bundler. Key state: `currentLang`/`wordLevel`/`currentCategory`/`currentSubcategory`/`categoryMode`, `grid` (2D letter array), `targetWords` (the words actually placed in the current puzzle — may be a subset of the category if some words didn't fit, see below), `found` (`[{word, cells}]`), `selection` (`[[r,c], ...]`, the in-progress tap sequence) and `direction` (locked after the 2nd tap of a new selection). `LEVEL_CONFIG` maps difficulty to `{minSize, maxSize, wordCount}` (easy 7–9/8 words, moderate 9–12/12, hard 11–14/16). Grid size is **not fixed per difficulty** — `buildPuzzle()` computes it from how many letters the chosen word-set's words actually contribute (`TARGET_FILL` targets ~50% raw fill) and clamps into that level's size band. This matters because sets vary in size (a category's smallest set can be well under `wordCount`): a fixed grid size per difficulty would make small sets look sparse on a big grid, which is exactly the "45 letters lost in a 169-cell grid" problem this whole density scheme exists to avoid. `DIRECTIONS` is the 8 straight-line vectors; word letters always sit in adjacent cells along one of these, so the tap-selection logic only ever needs to check immediate-neighbor steps, not arbitrary jumps.
- **`words.en.json`** / **`words.uk.json`** — `{ "categories": { "<id>": { "subcategories": [ { "id": "...", "name": "...", "sets": [ { "words": [...] }, ... ] }, ... ] } } }`. One file per language, 23 categories, and — per the "a category is a pure grouping node" rule above — **every** category has named subcategories now (e.g. `colors` splits into "Warm Tones" / "Cool Tones" / "Neutrals"; `animals` into "Farm & Pets" / "Wild Animals" / "Sea, Air & Reptiles" / "Baby Animals"); there's no longer a "plain category with bare `sets`" shape in the actual data (the code still tolerates one defensively, see `pickWordSet()`, but don't author new content that way). Each subcategory is named (its `name` field, shown in the category-select modal — *not* an `i18n.js` key, see below) and itself holds 1–3 curated **sets** — `pickWordSet()` in `script.js` picks one whole set at random per puzzle from the chosen subcategory, so a single game pulls from a thematically coherent group rather than the entire subcategory's full pool; sets themselves aren't named/exposed anywhere in the UI, just internal groupings within a subcategory. Words are lowercase, single alphabetic tokens (no spaces/hyphens), 3–16 letters (16 = the hard grid's max dimension), and each set mixes short (3–4 letter) and long (7+ letter) words — a set of same-length words makes the puzzle trivially easy regardless of grid size. Some subcategories are inherently narrow (`food`'s `softcheese`/`hardcheese`, `nativity`'s two, the `animals`' `seaair`/`baby`) and their sets run smaller (4–7 words) than the rest — that's fine, the dynamic grid sizing (see below) already shrinks the grid to match rather than producing a sparse puzzle; don't pad a narrow one with less-fitting words just to hit a round number. If you add a category, give it real subcategories from the start (not one bare `sets` array), keep that length spread, add its id to `CATEGORIES` in `script.js` **in label-alphabetical order** (see that array's comment — the category-select modal's top-level row order follows it), its data to both language files, and the matching `category<Id>` key to both `i18n.js` language objects — `renderCategoryTopList()` looks up `category${capitalize(id)}` and will render `undefined` if the key is missing. Note the sort is by the *English* label; `CATEGORIES` is one shared list used to render both languages' select lists, so the Ukrainian list follows English alphabetical order too, not Ukrainian — a known, accepted simplification rather than something to "fix" by maintaining two separately-sorted lists. Don't add an `i18n.js` key for a subcategory name — put it straight in each language's JSON `name` field instead (subcategory names, unlike category names, are per-language content, not UI chrome); subcategory ids must match exactly across `words.en.json` and `words.uk.json` (only the `name` differs) since language switches reuse `currentSubcategory` by id.

## Word placement & the puzzle generator (important limitation)

`buildPuzzle()` in `script.js` picks up to `wordCount` words from the active category (filtered to fit the grid) and sorts longer words first (they're harder to place). `placeWord()` places each word by **preferring overlap**: `findOverlapPlacements()` scans the grid for every already-filled cell matching one of the word's letters and tries anchoring the word through that cell in all 8 directions, keeping every placement that doesn't conflict with existing letters; if any such crossing placement exists, one is picked at random from those. Only when no word already on the grid shares a letter with it (always true for the first, longest word placed) does it fall back to `MAX_PLACEMENT_ATTEMPTS` random `(position, direction)` tries. This is what makes words actually interlock — don't revert to pure-random placement, since that's what caused the original "45 letters lost in a 169-cell grid" problem this scheme replaced. **A word that never finds a valid placement is silently dropped from that puzzle** — `targetWords` only ever contains words that actually made it into the grid, so the word-list panel and win condition are always consistent with what's actually findable. This is rare in practice but is a known, intentional trade-off rather than a bug — don't "fix" it by retrying the whole puzzle or increasing attempts indefinitely; that risks hangs on pathological inputs. If asked to guarantee every category word always appears, that requires a real backtracking placer (place-and-undo on failure), which is a bigger change — flag the trade-off rather than silently rewriting the generator.

**Crossings are capped per difficulty** via `LEVEL_CONFIG[level].maxCrossings` (easy: 2, moderate: 3, hard: 4) — how many words are allowed to share a single cell. A `crossCount` grid (same dimensions as the letter grid `g`, built fresh in `buildPuzzle()`) tracks how many words already pass through each cell; `fitsAt()`/`findOverlapPlacements()`/`placeWord()` all take `crossCount`/`maxCrossings` and `fitsAt()` rejects any candidate placement that would push a cell's count past the cap, so a would-be N+1-th crossing at an already-maxed cell is simply never offered as a candidate (the word instead looks for a different placement, or elsewhere in the fallback random-attempt loop). `buildPuzzle()` increments `crossCount` for every cell of a placement (target pass and bonus-duplicate pass alike) right after committing it to `g`.

Cell coloring for an overlap handles any number of crossing words, not just two: `overlapGradient(colorIndexes)` splits the cell into that many equal conic-gradient wedges (`360 / N` degrees each). This replaced an earlier `linear-gradient(135deg, a 50%, b 50%)` that only ever rendered a two-way split — a real bug, since it silently mislabeled genuine 3-way (or 4-way, on hard) crossings as a plain 2-way one instead of showing all the colors actually crossing there. If you touch cell coloring again, keep it N-aware; don't hardcode a 2-way assumption.

Empty cells are filled with a uniform-random letter from that language's alphabet (`ALPHABETS.en`/`ALPHABETS.uk`) — not frequency-weighted, so a grid can look a little more consonant-heavy or vowel-heavy than natural text. Same kind of documented simplification as `worder`'s difficulty-scoring caveat; flag it rather than silently trying to fix it if asked to make filler letters "more natural."

## Bonus words

**`bonusWords` is a subset of `targetWords`, not a separate list of extra words.** A bonus word is not something new to learn — it's one of the already-required words that this puzzle deliberately placed a *second* time somewhere else in the grid. Finding that second occurrence is the bonus; the word was already on the list either way.

`buildPuzzle()` places all of `chosen` (the required words) in their own pass first, longest-first, recording each word's first-placement cells in a `word -> cells` map as it goes. Only after that pass does it pick 2 or 3 words at random from the ones that actually got placed (`shuffle(placedTarget)`) and attempt a *second* `placeWord()` call for each, passing that word's first-placement cells as `excludeCells` so the second placement can't land on the exact same cells as the first (`placeWord(word, size, g, excludeCells)` filters candidates via `sameCellSet()`). A word only ends up in `bonusWords` if this second placement actually succeeds — same "may be dropped" caveat as everywhere else in this file; on a tight grid a word might simply not get a second spot, and that's fine, it just isn't a bonus word this round.

At runtime the two occurrences are indistinguishable by text — `selectedWord()` only produces a string, so `checkMatch()` has to reason about *which cells* were traced, not just which word:
- First time a word's text is completed (`found.find(f => f.word === text)` misses): normal find, pushed to `found`, counts toward the win condition as always.
- Second time the same text is completed: if the traced cells are the exact same set as the recorded find (`sameCellSet()`), it's just a retrace — no-op, don't award anything twice for re-tracing what's already found. If the cells are genuinely different **and** this word is one of this puzzle's `bonusWords` **and** it hasn't already been bonus-credited, push it to `bonusFound` and call `recordBonusFound()`.

Finding the bonus occurrence marks that same word's chip with `★ word` and `.bonus-chip-found` styling (gold background, matching `.cell.bonus` on the grid) in place of the normal `word-color-N` — `renderWordList()` iterates `targetWords` exactly once per word (there's no separate bonus chip list), checking `bonusFound` before `found` since a bonus find implies a normal find already happened. Bonus finds have **no effect on the win condition** (still exactly `found.length === targetWords.length`) and increment `bonusWordsFound` immediately, not gated by `statsRecorded`. The header's `#found-badge` (`updateFoundBadge()`) shows bonus progress alongside the normal count when this puzzle has any bonus words at all — `"3/9 ★1/2"` — since a bonus find never touches `found.length`/`targetWords.length` and would otherwise be invisible there; `checkMatch()`'s bonus branch calls `updateFoundBadge()` itself, same as the normal-find branch does.

(This design replaced an earlier one where `bonusWords` were 2-3 *extra* words drawn from the rest of the category pool, placed once each, separate from `targetWords` — rejected because the point of a bonus is spotting a repeat of something already required, not learning an unrelated extra word. If you touch this again, don't reintroduce a separate bonus word pool.)

Don't be surprised if a bonus word is rare in a particular playthrough — it depends entirely on chance letter arrangement, there's no guarantee one appears in every puzzle. If asked to make bonus words appear more reliably, that would mean deliberately placing extra words during generation (a real feature change to `buildPuzzle()`/`placeWord()`), not something to fake by lowering the bar for what counts as a bonus match.

## Selection interaction (why it's implemented this way)

The selection state machine — `handleCellTap(r, c)` — is the single source of truth for what tapping/dragging over a cell does, regardless of how that cell got activated:

1. First activation on any cell starts `selection = [cell]`.
2. The second activation only extends the selection if it's an **immediate 8-directional neighbor** of the first cell — that locks `direction` for the rest of this selection. A non-adjacent second activation discards the first cell and starts a fresh selection at the new cell instead (no error state, just a restart).
3. From then on, only the exact next cell in the locked direction (`last + direction`) extends the selection; anything else restarts a fresh selection there.
4. Re-activating the *current last* selected cell pops it (undo one letter) — this single rule also handles "clear everything," since popping the only cell in a length-1 selection empties it. There is intentionally no separate "activate the first cell to clear all" rule; keep it that way unless asked to change it, since a second rule here would create an ambiguous case when the first and last cell are the same.

After every activation, `checkMatch()` joins the selected cells' letters and compares the string directly against `targetWords` still unfound — a match is a match regardless of which direction it was read in, so there's no separate forwards/backwards check.

**Drag support (mouse + touch) reuses this exact same state machine, one activation per newly-entered cell — it is not a second, parallel selection system.** `handlePointerDown`/`handlePointerMove`/`handlePointerUp` (Pointer Events, so mouse/touch/pen all go through one code path) detect a press-and-move gesture and call `handleCellTap(r, c)` once per cell the pointer newly enters, in order — so dragging across a whole word produces exactly the same `selection`/`direction` progression as tapping each of those cells individually would. This is what makes "drag partway through a word, then finish it with plain taps" (or the reverse) work with zero special-casing: neither mode owns the selection, they're just two ways of feeding the same `handleCellTap` calls.

`cellFromPoint()` computes a cell from continuous coordinates (`((x - rect.left) / rect.width) * gridSize`, etc.) against `#board`'s own bounding rect, not a DOM hit-test — `document.elementFromPoint()` was tried first and dropped: it returns nothing useful over `#board`'s inter-cell gap pixels, and more importantly gives zero tolerance for the pointer drifting off a cell's exact box. It's deliberately dumb and immediate: the plain nearest cell, every call, no threshold, no direction-awareness — selection happens *on hover*, not after a delay. All the smarts live in `handlePointerMove()` (below), not here.

**Diagonal dragging went through two failed designs before landing on this one, and both failure modes are worth knowing if you touch this again.**

Diagonal neighbors share only a single corner point, while orthogonal (horizontal/vertical) neighbors share a full edge — so a diagonal drag's pixel path routinely clips through the corner of an unrelated orthogonal neighbor's cell box on the way to the intended diagonal cell. This is real grid geometry, not jitter, and it happens *both* between letter 1→2 (establishing direction) and letter 2→3 (the first extension after it's established) — a fix that only covers one of those two transitions will still "mostly" fail on diagonals.

- **First attempt**: derive the direction from the overall displacement vector since `pointerdown`, only once it exceeded a small threshold (`DIRECTION_LOCK_THRESHOLD`). This actually worked (~95-100% success under synthetic wobble testing) — but it delayed the *very first* selection response by that threshold, which read as "hovering over letters doesn't select them" / "selection prevented" to a real user, since drag-select is expected to respond instantly to hover. Reported and reverted.
- **Second attempt**: pick whichever cell is nominally nearest the pointer on every hover, immediately, with no threshold, and reset the selection to a fresh start on any hover that isn't adjacent to letter 1 (mirroring a tap's "unrelated cell" rule). This restored instant response but reintroduced heavy diagonal failures (~50% in testing) — the corner-clip cell is typically two cells from letter 1, not adjacent to it, so it kept tripping the "unrelated, reset everything" rule for both the 1→2 and 2→3 transitions.
- **Current design**: instant hover response (no threshold, matching what a drag-select is expected to feel like) *and* corner-clip immunity, by treating "ambiguous" as "wait for the next hover" rather than "reset":
  - At `selection.length === 1` (no direction yet): a hover adjacent to letter 1 is applied immediately as letter 2. A hover that *isn't* adjacent (the corner-clip case) is simply ignored — not reset, not applied — leaving letter 1 selected until a genuinely adjacent hover arrives.
  - At `selection.length === 2` (letter 2 already picked, provisionally): a hover matching `last + direction` confirms letter 3 and fully locks the drag from here on (see below). A hover back on letter 1 undoes the provisional letter 2. A hover on a *different* neighbor of letter 1 than the current pick is a genuine live correction — swap to it (undo the old pick, apply the new one) — this is what makes hovering near a diagonal letter always select it, even if an earlier hover briefly landed on the wrong neighbor. Anything else (the corner-clip case again, now between letter 2 and letter 3) is ignored, same as above.
  - At `selection.length >= 3` (direction fully confirmed): `isValidDragContinuation(r, c)` only lets a hover apply if it's the exact next cell (`last + direction`) or the undo-last case; anything else is ignored. The drag coasts through any remaining corner-clip ambiguity until the pointer reaches the real next cell.

The unifying rule across all three phases: **an ambiguous hover is something to wait out, never something to reset over.** The only place a hard reset still happens is a brand new `pointerdown` (or a hover exactly on letter 1 with no provisional letter 2, which is just a no-op, not a reset) — redirecting an in-progress drag to an unrelated, non-adjacent cell now requires lifting and starting a new gesture, which was judged an acceptable trade for diagonal reliability. If you touch this again, don't reintroduce "reset selection on any non-matching hover" — that's the exact design that made diagonals fail in testing.

A plain tap (pointerdown + pointerup with no cell change in between) is applied from `handlePointerUp`, not from a "click" listener — **this was a real shipped bug**: relying on the browser's native trailing `click` event to know when a drag had ended (and to suppress a duplicate re-application of the last-dragged cell) failed unpredictably, because whether/when that `click` fires varies by pointer type and gesture; the observable symptom was the *next* genuine tap after a drag silently doing nothing, because a stale suppression flag was still armed waiting for a click that either never came or came at the wrong moment. The fix: `pointerup` itself decides and acts — apply the tap directly if the pointer never left its starting cell, do nothing further if it did (the drag already applied every cell via `handlePointerMove`) — and only use the native `click` listener as a fallback for keyboard-activated buttons (Tab + Enter/Space), which never generate pointer events at all. `suppressNextClick` (with a 300ms timeout safety net, since a real click sometimes never arrives at all after a touch drag) exists purely to swallow whatever trailing `click` *does* show up after a pointer-driven interaction, so it can't double-apply. If you touch this again, keep `pointerup` as the actual source of truth for taps; don't move that logic into `click`.

**A word can be completed mid-drag, finger/button still down** — `checkMatch()` clears `selection`/`direction` on a match regardless of what triggered it. `handlePointerMove` explicitly handles `selection.length === 0` by re-anchoring the still-active drag at the pointer's current cell (a fresh `dragStartCell`/`dragStartClientX`/`dragStartClientY`, exactly like a new `pointerdown`) rather than falling through to `isValidDragContinuation()`, which assumes a non-empty `selection` and would throw reading `selection[-1]`. This also means one continuous drag can find several words back-to-back without lifting.

`.cell` has `touch-action: none` so a touch drag across the grid selects letters instead of scrolling the page.

**Badge rendering — why it's a separate overlay, not a child of the cell.** Badges float outside their cell's corner (that's the intended look), which means they visually overhang into whichever cell is next to them. Cells don't have distinct z-index tiers, so if a badge were a child of its own cell, two *adjacent selected* cells (both then sharing the same effective stacking level) would fight over paint order — and browsers break that tie by DOM order, which for a grid always runs left-to-right/top-to-bottom regardless of tap order. In practice that meant the left (or top) cell's badge got clipped by its neighbor for horizontal/vertical selections specifically — an actual shipped bug, not a hypothetical. The fix: `#badge-layer` is a sibling of `#board`, absolutely positioned over the exact same box with the same `--cols` grid geometry (both read `--cols` off their shared parent `#board-wrap`, since a custom property set on `#board` itself wouldn't reach a sibling). `renderSelection()` places each badge with inline `grid-row`/`grid-column` matching its cell's `(r, c)`, then CSS (`justify-self: end; align-self: start; transform: translate(45%, -45%)`) pushes it to that grid cell's corner and out past its edge. Because the whole layer paints above the whole board in one pass, no per-cell stacking fight is possible. `#badge-layer` has `pointer-events: none` so clicks pass through to the actual cell buttons underneath. Don't move badges back to being per-cell children to "simplify" this — that's what caused the bug.

## Start screen & pause (timer model)

A freshly generated puzzle does **not** start ticking immediately — `generateNewPuzzle()` puts it straight into the paused state (`setPaused(true)`) so it opens on a "Ready to play" / Start screen, board and word list hidden, exactly like a mid-game pause. This is deliberate unification, not two features bolted together: **starting is just resuming from zero.** `togglePause()` is the one function that both starts and pauses/resumes; `setPaused()` decides which copy to show (`readyLabel`/`startBtn` vs `pausedLabel`/`resumeBtn`) purely from `hasStarted()` (`pausedElapsedMs > 0 || startTime !== null || found.length > 0`) — there's no separate "first tap starts the timer" code path anymore (there used to be, in `handleCellTap()`; removed on purpose, don't re-add it, it fights with the pause overlay covering the board anyway). `hasProgress()` (the confirm-modal guard for New Game / category / difficulty / language switches) deliberately checks `hasStarted()` rather than `paused` alone — a freshly generated, never-started puzzle is technically "paused" but has nothing to lose, so it shouldn't trigger the confirm dialog.

Elapsed time itself is banked in two parts: `pausedElapsedMs` (time from all earlier segments) plus, only while a segment is actively ticking, `Date.now() - startTime`. `startTime` is `null` whenever nothing is actively ticking — before Start is tapped, while paused, or (deliberately) right after a reload that landed on a paused puzzle. `elapsedMs()` always returns `pausedElapsedMs + (startTime ? Date.now() - startTime : 0)`, so every other piece of code (the win message, stats, `saveState()`) just calls `elapsedMs()`/`formatTime()` without needing to know whether the game is currently paused. `togglePause()` is the only place that moves time between the two buckets: pausing folds the current segment into `pausedElapsedMs` and clears `startTime`; resuming (or starting) begins a fresh segment (`startTime = Date.now()`) on top of the existing `pausedElapsedMs`. `updateTimerDisplay()` is called immediately on pause (not left to the next 1-second interval tick) so the frozen number is exact, not up to 999ms stale.

The pause/start UI (`#pause-overlay`, `.paused` classes on `#board-wrap`/`#word-list`) is purely a visual + interaction block — `handleCellTap()` no-ops while `paused`, and the overlay physically covers the board so clicks can't reach the hidden cells underneath anyway; clicking anywhere on the overlay resumes/starts. `paused` is persisted (`wasPaused` in `saveState()`) specifically so a reload mid-pause (or a reload of a never-started puzzle) comes back the same way — overlay up, board still hidden, timer still frozen at the same value — instead of silently revealing the board. Don't drop that field to "simplify" persistence, that's the one case this whole timer-banking scheme exists to get right.

## Category selection (Random vs. Select)

The menu doesn't list categories directly anymore — with dozens of categories it became an unusable wall of buttons. Instead, the "Category" menu section is a 2-way toggle, `categoryMode` (`"random"` / `"select"`), and what it controls is **what New Game does**, not what's currently loaded:

- **`random`** (default): `startNewPuzzleFlow()` calls `pickRandomCategoryAndSubcategory()` — a random entry from `CATEGORIES`, then a random one of *its* subcategories (the synthetic `"mixed"`/All Mixed option is included in that pool, not special-cased out) — then generates immediately.
- **`select`**: `startNewPuzzleFlow()` calls `openCategorySelectModal()` instead of generating anything. The picker is a **single-screen tree** (`renderCategoryTree()`), not a second screen you navigate into — three earlier designs were tried and replaced: a flat "Category — Subcategory" list (read as cluttered mixing one-row categories with Animals' five rows), a two-step "categories, then a separate subcategories screen with a Back row" (the full-screen swap-out felt like an unwanted extra window), and a tree where tapping the category **label** itself immediately started an "All Mixed" puzzle while a separate ▸ toggle expanded it (this made the label ambiguous — sometimes a pick, sometimes just a browse gesture, depending on whether you'd looked at the topics yet). The current version: **the label and the ▸ toggle do the exact same thing** — both just expand/collapse that category's subcategories indented underneath, in place, no exceptions. Only tapping a specific named subcategory row, or an explicit **"All Mixed"** row appended at the end of every expanded list (`tree-mixed-row`, `subcategoryId = "mixed"`), calls `generateNewPuzzle()` and closes the modal — there is no shortcut that skips seeing the topic list first. **No puzzle is generated until one of those rows is tapped** — this is why `startNewPuzzleFlow()` and `generateNewPuzzle()` are separate functions; conflating them would mean Select mode couldn't defer generation.

  A **"🎲 Random" row** (`tree-random-row`) is appended once at the very end of the whole tree, after every category — it calls the exact same `pickRandomCategoryAndSubcategory()` that Random *mode* uses, then `pick()`s the result immediately. This is a one-off convenience action inside the Select picker, **not** a way to switch to Random mode: it doesn't touch `categoryMode` or `CATEGORY_MODE_KEY` at all, so the next New Game still opens this same picker rather than silently starting to auto-generate. If you're tempted to "simplify" this by having it just call `startNewPuzzleFlow()` or by setting `categoryMode = "random"` — don't; both would change the user's persisted preference as a side effect of a single tap, which is exactly what this row is designed not to do.

Difficulty and language changes deliberately do **not** go through `startNewPuzzleFlow()` — they call `generateNewPuzzle(currentCategory, currentSubcategory, ...)` directly, keeping whatever category/subcategory is already loaded. Only New Game (and a fresh app load with no saved puzzle) re-rolls or re-prompts for category. Don't route difficulty/language changes through the random-or-select flow — that would force a Select-mode user to re-pick a category just to bump the difficulty, which isn't what "change difficulty" should do.

`pickWordSet(category, subcategoryId)` is the single place that resolves a category/subcategory pair down to a word pool: it branches on whether `WORDS[category]` has `.subcategories` (look up by id, or union every subcategory's words into a deduplicated `Set` for `"mixed"`) or plain `.sets` (a fallback for a category with no subcategories — currently vestigial, since every category has subcategories now, but kept as a defensive branch rather than assumed-impossible). Keep `buildPuzzle()` and everything downstream category-shape-agnostic by only ever going through this function — don't reach into `WORDS[category].sets` directly elsewhere.

**A category is a pure grouping node — it never holds words directly, only subcategories.** This was a deliberate correction: `cheeses` and `fruit` used to be their own top-level categories sitting right next to `food`, which is backwards (cheese and fruit *are* foods) — they're now `food`'s `softcheese`/`hardcheese`/`fruit`/`exoticfruit` subcategories instead (`food`'s original catch-all "Vegetables & Produce" set was also split to remove the fruit words it had accidentally absorbed, avoiding a same-word duplicate between `food`'s own vegetables subcategory and the newly-merged fruit subcategories). If a new category idea is really a topic *within* an existing category (the way Cheeses was within Food), add it as a subcategory of that category, not a sibling of it — don't reflexively default to a new top-level entry in `CATEGORIES`.

Known edge case, not fully solved: the category-select modal doesn't block the rest of the page (it's a normal `.modal`, and `#menu-panel` has a higher `z-index`), so a user could open the "⋮" menu and change difficulty/language while the modal is still open and no puzzle has ever been generated yet (`currentCategory` still at its unset default). That's a narrow, rare interaction — flagged here rather than engineered around, since properly blocking it would mean disabling header controls while the modal is open, which isn't worth the complexity for how unlikely the sequence is.

**The tree highlights whatever category/subcategory the puzzle on screen is actually using**, and pre-expands (and scrolls to) that category, so reopening the picker mid-game shows what's currently active instead of always coming up fully collapsed — see `isCurrentCategory`/`isCurrentSub` and the `.selected`-class/`.tree-subcategories` (no `hidden`) branches inside `renderCategoryTree()`. This is derived fresh from `currentCategory`/`currentSubcategory` every time the modal opens; there's no separate "last picked" state to keep in sync.

**Difficulty is duplicated inside the category-select modal too** (`#category-select-level-options`, rendered by `renderCategorySelectDifficulty()`), so picking a category and setting the difficulty for it happens in one place instead of round-tripping through the "⋮" menu separately. This is why `startNewPuzzleFlow()`/`renderCategoryTree()`/`generateNewPuzzle()` calls all read `wordLevel` fresh rather than threading a `level` parameter through — a parameter captured when the modal opened would go stale the instant this in-modal picker changed it. The in-modal picker's `onSelect` calls `renderMenus()` after updating `wordLevel` specifically to keep the "⋮" menu's own difficulty radio in sync — the two pickers share the same `wordLevel` state and `LEVEL_KEY` persistence, they're two views of one value, not two independent settings. If you add a third place that can change difficulty, make sure it re-renders (or is re-rendered by) both.

## State & persistence

`localStorage` keys, mirroring `worder`'s naming style:

- `wordweave-lang` — active language (`"en"`/`"uk"`), detected via `navigator.language` if unset.
- `wordweave-level` — active difficulty (`"easy"`/`"moderate"`/`"hard"`), defaults to `"moderate"`.
- `wordweave-categorymode` — `"random"` or `"select"` (see "Category selection" above), defaults to `"random"`. There's deliberately no persisted "last category" preference anymore — Random mode always re-rolls, Select mode always re-prompts; the *currently loaded* category/subcategory live only in the per-language puzzle state below.
- `wordweave-state-${lang}` — the full in-progress puzzle: `category`, `subcategory` (the id, `"mixed"`, or `null`), the *generated* grid, `targetWords`, `bonusWords` (see "Bonus words" above — this is the designated list of which words are bonus, persisted since it's decided once at generation time, not recomputed on load), `found`, `bonusFound`, `gameOver`, and a timer snapshot (`elapsedMsAtSave`, `timerWasRunning`, `wasPaused`) so reload resumes the exact puzzle and keeps the clock running (or stays paused) rather than replaying a seed. Keyed per language, like `worder`.
- `wordweave-stats-${lang}` — `{ played, completed, wordsFound, bonusWordsFound, bestTimes: {easy, moderate, hard} }`. `played` increments every time a new puzzle is generated (including on first load); `completed`/`wordsFound`/`bestTimes` only update once, guarded by `statsRecorded`, when all of a puzzle's `targetWords` are found; `bonusWordsFound` increments immediately on each bonus find, independent of `statsRecorded`.

If you add new persisted fields, update both the save (`saveState`/`saveStats`) and load (`loadState`/`loadStats`) functions, and give the load side a sane default so old saved data (missing the new field) doesn't break — `loadStats` already does this per-field.

## Responsive layout (portrait vs. landscape)

Portrait (and narrow/short landscape) stacks `#board-column` (board + message) above `.word-list`, same as always. On wide landscape viewports (`@media (orientation: landscape) and (min-width: 700px)`, the width gate exists so short landscape phones — little vertical room, not much width either — stay in the stacked layout instead of squeezing everything sideways), `main` switches to a row: board on one side, `.word-list` as a scrollable side column on the other. `#board-wrap`'s width formula also switches from `min(92vw, 520px)` to `min(70vh, 480px)` in that mode, since height is now the constraining dimension, not width.

**This media query block lives at the very end of `style.css`, after every rule it overrides — keep it there.** `#board-wrap` and `.word-list` both have unconditional base rules earlier in the file that set the same properties (`width`, `flex-direction`). CSS resolves equal-specificity conflicts by *source order*, not by "media query beats plain rule" — a base rule appearing later in the file wins over an earlier media-query override for the same selector/property, media query match or not. This shipped broken once already (the landscape layout silently no-op'd, board rendered at the wrong width and pushed off-screen) before being moved down; don't move it back up "for readability" without re-testing at a landscape viewport width.

Also note: `.word-list`'s base rule sets `flex-wrap: wrap`. The landscape override sets `flex-direction: column` but must also set `flex-wrap: nowrap` — `column` + inherited `wrap` together, once content exceeds `max-height`, wrap into *additional columns* instead of scrolling one column, which silently broke the side-list layout the same way (chips overflowing sideways in pairs) even after the source-order fix above.

**Cell letter size.** `.cell`'s `font-size` is `clamp(0.6rem, calc(var(--cell-size, 32px) * 0.5), 1.5rem)`, where `--cell-size` is set on `#board-wrap` in JS (`updateCellFontSize()`, called after every `renderBoard()` and on a debounced `window resize` listener) from the *actually rendered* cell box (`boardWrapEl.getBoundingClientRect().width / gridSize`). Don't go back to a `vw`-based formula for this — that was tried first and looked fine in landscape purely by accident (there, `vw` happens to track the board's long axis) but was too small in portrait, where `vw` is the narrow axis and has no relationship to the actual cell size once you factor in `--cols` (9–14 depending on difficulty) and the landscape/portrait width formulas being completely different (`min(92vw, 520px)` vs `min(70vh, 480px)`). Measuring the real box is what makes the ratio (font ÷ cell width, ~0.52–0.54) stay consistent across every orientation × difficulty combination instead of guessing.

- No comments explaining *what* code does — only *why*, when non-obvious (see existing sparse comments as the bar).
- No build tooling, no dependencies. Keep it that way unless explicitly asked to add a bundler/framework.
- Overflow actions live in the "⋮" menu (`#menu-panel`), not as standalone header buttons — reuse the existing `radiogroup` pattern (`buildRadioOptions()`) for any new global setting rather than adding a header button.
- Native browser dialogs (`confirm()`/`alert()`) are intentionally avoided in favor of the styled `.modal` pattern — don't reintroduce them. The `pendingConfirmAction` + `requestFreshGame()` guard (shared by New Game, and by difficulty/language changes) is the pattern to reuse for anything else that should warn before discarding in-progress puzzle state.
- Each found word gets its own color from an 8-color round-robin palette (`--word-0`..`--word-7` in `style.css`), assigned by `wordColorIndex()` in `script.js` from the word's fixed position in `targetWords` — stable across re-renders and reloads without needing extra persisted state. A cell shared by two found words renders a diagonal two-color split (`linear-gradient`, built inline in `renderBoard()` from `var(--word-N)` so it stays theme-aware); three+ overlapping found words at one cell only show the first two colors (rare, undocumented beyond this note — a real fix would need a wedge/stripe layout per additional color).

## Offline support (service worker)

`sw.js` is a cache-first service worker, same pattern as the sibling `worder` project. `PRECACHE_URLS` lists every file the app needs (markup, styles, script, both languages' `i18n.js`/`words.*.json`, icons, manifest); the `install` handler caches all of them up front so the app works with zero network connection after the first visit, not just for previously-visited puzzles. `registerServiceWorker()` in `script.js` registers it on `window load` and force-reloads once when a new service worker takes control (`controllerchange`), so an update actually replaces the running page instead of leaving stale JS active in a tab that's never closed.

**If you add a new site asset** (a new language file, a new icon size, anything `index.html` references), add it to `PRECACHE_URLS` in `sw.js` too — anything not precached still works online (the fetch handler falls through to the network and caches the response opportunistically) but won't be available offline until it's been fetched at least once.

Cache invalidation: `CACHE_NAME` embeds `CACHE_VERSION`, a `__CACHE_VERSION__` placeholder that `.github/workflows/pages.yml` replaces with `$GITHUB_SHA` at deploy time (`sed` step, mirrors the file-copy `cp` list — `sw.js` has to be in both). A new commit means a new cache name, and the `activate` handler deletes every other cache, so old assets can't linger. Locally the placeholder is never substituted (no CI step runs), so local testing always uses one cache named literally `wordweave-__CACHE_VERSION__` — that's expected, not a bug; it still round-trips through the real precache/fetch logic correctly, it just doesn't demonstrate the invalidation-on-deploy behavior (verify that on an actual deploy, or by manually editing the placeholder).

Cache lookups use `{ ignoreSearch: true }` (`caches.match(event.request, { ignoreSearch: true })`) so a request for `style.css?v=11` still matches the precached `style.css` entry — `index.html` bumps a manual `?v=N` query param on `style.css`/`script.js` for plain-HTTP-server local dev cache-busting (see "Running & verifying changes" above), and without `ignoreSearch` those two cache-busting mechanisms would fight each other (every version bump would look like a cache miss to the service worker too, which is harmless but defeats offline-availability until a fresh network fetch happens).

**`registerServiceWorker()` skips registration entirely on `localhost`/`127.0.0.1`** (and actively unregisters/clears caches from any *earlier* local registration). This was a real bug, not a preemptive guard: `CACHE_NAME`'s version placeholder is only ever substituted by the deploy workflow, so a service worker registered while testing locally would cache-first-serve one permanent snapshot of the app forever — every subsequent local edit, and every `?v=N` cache-bust, silently ignored, because `ignoreSearch: true` (needed for production, see above) means the SW doesn't even look at the query string to notice something changed. Whoever hit this saw "the local version is old" with no obvious cause, since the *files on disk* and the *server* were both already correct — the browser's installed service worker was the actual stale layer, invisible unless you know to check DevTools → Application → Service Workers. If you ever need to test the *offline* behavior locally, temporarily comment out the localhost-skip block — don't remove it permanently, since that reintroduces the stale-cache trap for every future local session.

## Deployment

`.github/workflows/pages.yml` deploys `main` to GitHub Pages on every push, serving at the custom domain in `CNAME` (`wordweave.shadoll.com` — DNS for this must be configured separately from this repo). The workflow copies an explicit file list into a `dist/` folder before publishing — if you add new site assets (a new language's word file, a new icon size, etc.), add them to that `cp` list too or they won't ship.
